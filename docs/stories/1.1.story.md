# Story 1.1 — Setup Firebase: Firestore + Security Rules

## Status

Ready

## Story

> Como fundador do Trivum,
> quero as collections do Firestore criadas com Security Rules corretas,
> para que dados de leads sejam persistidos com segurança desde o primeiro formulário.

## Acceptance Criteria

- [ ] **AC1:** Collections `leads`, `lead_notes`, `products` criadas no Firestore
- [ ] **AC2:** Security Rules configuradas:
  - `leads`: write público (criação sem auth); read e update apenas autenticados
  - `lead_notes`: CRUD apenas autenticados
  - `products`: CRUD apenas autenticados
- [ ] **AC3:** Firebase Admin SDK configurado em `lib/firebase/admin.ts` (server-only)
- [ ] **AC4:** Firebase Client SDK configurado em `lib/firebase/client.ts` (browser)
- [ ] **AC5:** Variáveis de ambiente documentadas em `.env.example` com todos os valores necessários
- [ ] **AC6:** `apphosting.yaml` atualizado com referência às env vars de Firebase

### Integration Verification

- [ ] **IV1:** Build de produção `next build` passa sem erros após adição dos SDKs Firebase
- [ ] **IV2:** Firebase App Hosting não conflita com nova configuração de Admin SDK (server-only)
- [ ] **IV3:** Landing page existente renderiza sem alteração após setup

## Scope

### IN
- Criação das 3 collections no Firestore (estrutura via Security Rules e documento inicial de teste)
- Firebase Security Rules completas para `leads`, `lead_notes`, `products`
- `lib/firebase/admin.ts` — inicialização do Admin SDK com `firebase-admin`
- `lib/firebase/client.ts` — inicialização do Client SDK com `firebase/app`, `firebase/firestore`, `firebase/auth`
- `.env.example` com todas as env vars necessárias (Firebase + placeholders para Resend)
- Atualização de `apphosting.yaml` com referência às env vars

### OUT
- Implementação de endpoints de API (Story 1.2)
- Componentes de UI ou formulário (Story 1.3)
- Firebase Auth (Story 1.5)
- Seed de dados reais no Firestore
- Configuração de Firebase Emulator Suite (pode ser feito como melhoria futura)

## Dependencies

- **Pré-requisito:** Nenhum — esta é a Story fundação (1.1 bloqueia todas as demais)
- **Bloqueada por:** Nenhuma
- **Bloqueia:** Stories 1.2, 1.3, 1.4, 1.5, 1.6, 1.7, 1.8, 1.9, 1.10, 1.11

## Complexity Estimate

**Pontos:** 3 (Small)
- Setup repetível, padrão bem documentado
- Sem lógica de negócio
- Risco principal: conflito Admin SDK + Firebase App Hosting (IV2)

## Business Value

Fundação obrigatória do sistema operacional Trivum. Sem Firestore configurado com Security Rules corretas, nenhum lead pode ser persistido com segurança. Desbloqueador crítico do pipeline completo.

## Risks

| Risco | Impacto | Mitigação |
|---|---|---|
| Admin SDK conflita com Firebase App Hosting runtime | Alto | Usar `firebase-admin` apenas em server contexts; não importar em Client Components |
| Security Rules mal configuradas expõem PII | Alto | Testar regras com Firebase Rules Simulator antes de commitar |
| Env vars não carregam no App Hosting | Médio | Validar `apphosting.yaml` contra documentação oficial |

- [ ] Todos os ACs e IVs acima passam

### Task 2 — Configurar Firebase Admin SDK (AC3)

### Task 3 — Configurar Firebase Client SDK (AC4)
- [ ] 3.1 Criar `lib/firebase/client.ts`
- [ ] 3.2 Inicializar `firebase/app` com variáveis de ambiente públicas: `NEXT_PUBLIC_FIREBASE_API_KEY`, `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`, `NEXT_PUBLIC_FIREBASE_PROJECT_ID`
- [ ] 3.3 Exportar `clientDb` (Firestore) e `clientAuth` (Auth) inicializados
- [ ] 3.4 Usar padrão singleton para evitar múltiplas inicializações: `getApps().length ? getApp() : initializeApp(config)`

### Task 4 — Criar Security Rules no Firestore (AC1, AC2)
- [ ] 4.1 No Firebase Console, acessar Firestore > Rules
- [ ] 4.2 Implementar regras conforme spec do PRD:
  ```
  rules_version = '2';
  service cloud.firestore {
    match /databases/{database}/documents {
      // leads: write público, read/update autenticado
      match /leads/{leadId} {
        allow create: if true;
        allow read, update: if request.auth != null;
        allow delete: if false;
      }
      // lead_notes: CRUD apenas autenticado
      match /lead_notes/{noteId} {
        allow read, write: if request.auth != null;
      }
      // products: CRUD apenas autenticado
      match /products/{productId} {
        allow read, write: if request.auth != null;
      }
    }
  }
  ```
- [ ] 4.3 Testar no Firebase Rules Simulator: (a) create lead sem auth → ALLOW; (b) read lead sem auth → DENY; (c) read lead autenticado → ALLOW
- [ ] 4.4 Publicar as regras

### Task 5 — Documentar env vars (AC5, AC6)
- [ ] 5.1 Criar/atualizar `.env.example` com as variáveis abaixo (sem valores reais):

- Env vars: `.env.example` (novo), `.env.local` (gitignored)

### Firestore Collections
Três collections definidas pelo PRD [Source: docs/prd.md#4.2]:
- `leads` — dados de candidatura dos especialistas
- `lead_notes` — notas e interações append-only
- `products` — portfólio de produtos do studio

### SDK Split (Critical)
Firebase usa dois SDKs com contextos distintos [Source: docs/prd.md#4.2]:
- **Admin SDK** (`firebase-admin`): server-only. Usado em Route Handlers e Server Components. NUNCA importar em Client Components.
- **Client SDK** (`firebase/app`, `firebase/firestore`, `firebase/auth`): browser. Usado em Client Components para operações autenticadas diretas (ex.: kanban Story 1.6).

### Security Rules Pattern
Regra crítica do PRD [Source: docs/prd.md — Story 1.1 AC2]:
- `leads.create`: público (sem auth) — necessário para formulário público (Story 1.3)
- `leads.read/update`: autenticado — protege PII dos especialistas
- `leads.delete`: bloqueado — sem operação de delete via regras (apenas Admin SDK server-side)
- `lead_notes` e `products`: auth obrigatório em todo CRUD

### Compatibility Constraint
PRD exige [Source: docs/prd.md#CR2]: "Adição de Firestore/Auth não conflita com configuração existente do Firebase App Hosting (`apphosting.yaml`)". Validar build de produção é IV1 obrigatório.

### Testing Requirements
Sem testing-strategy.md disponível. Strategy mínima para esta story:
- Firebase Rules Simulator (no console) para AC2
- `next build` para IV1
- Inspeção manual de imports para IV2
